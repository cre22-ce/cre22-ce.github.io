Artifacts and Research:

1. A Program

My first artifact is a program that I have written that encodes and decodes Portable Network Graphic (png) files. There are 2 files associated with this: Cargo.toml and lib.rs. Each file is explained below. This is an example of pure code that a Computer Engineer would have to write.

This code is written in a language called Rust, a memory safe, high performance programming language. ?

1.1 Cargo.toml:

This manifest file tells the compiler (the program that turns human readable code into instructions for the processor) information about the binary (the executable file generated by the compiler; a.k.a application, program) and the required dependencies (external bundles of code used in this program; a.k.a libraries, packages, crates !!) that need to be included.

This specific file is telling the compiler that:

  1. The package name is "png"
  2. This is version "0.1.0"
  3. The list of authors are ["Caleb Eastman"] (only one author in this list)
  4. The edition of this program is "2018"
  5. This program uses the flate2 library


1.2 lib.rs:

This source file contains all the code specific to this application. At the top of the file (though it can be anywhere), the libraries that were listed in Cargo.toml are imported into the scope (a collection of variables and functions that can be accessed) of this program. The std library means "standard" and is included in the compiler so it doesn't need to be listed separately but needs to be requested explicitly here to use it. The text after the :: indicates that the program only uses that specific part of the library instead of the whole thing, cutting down on binary size and unused code in the end.

This code was in the middle of a complete rewrite when I stopped working on it so I'm not even sure this works properly. Luckily, this shows how a Computer Engineer might organize their workflow in the middle of a project. Lines that start with // or are surrounded by /* ... */ are comments that are ignored by the compiler. Comments serve as an easy way for programmers to communicate with other programmers who might use their code or as a way to communicate with themselves in the future. Most of the comments in this file are notes to myself on what a particular line of code does or, if a line starts with //TODO: ("TODO:" gets highlighted in my text editor so it stands out), it would tell me what my next step in development should be.



2. Documentation

My second artifact is a documented file. While it may seem like just another source file, this has some special comment lines in it. Before I said that any line that started with // was a comment, but there is a special case for lines that start with ///.
This is called a documentation line and provides both the integrated development environment (IDE) ? and the programmer with some critical information. Documentation lines are basically mini technical write-ups about the function and use of individual lines of code. They are placed above the lines they intend to document. These lines are important for the next part of this artifact.

This specific code is a utility that parses binary HTTP Responses and HTTP Requests. The data is stored in objects that can be mutated that can then be turned back into binary to send through an HTTP socket.

Note: I omitted the Cargo.toml file for this artifact as it is more of the same; The libraries included are regex and lazy_static.
